## 📚 零基础速查：本文档会反复用到的 JS / TS 语法

> 如果你 JS / TS 基础不太好，**先把这一节看一遍**。下面每个文件里出现的写法，几乎都能在这里找到对应解释。后面每段代码的「逐行解释」会默认你已经认识这些基础语法。

### 一、TypeScript 是什么

TypeScript（TS）= JavaScript（JS）+ **类型标注**。
浏览器/Node 最终运行的还是 JS，TS 只是在你写代码时多了一层「类型检查」，能在**写代码阶段**（而不是上线后）就发现 “你把数字当字符串用了” 这类错误。`.ts` / `.tsx` 文件经过编译会变回 `.js`。

- `.ts`：纯逻辑文件（不含 JSX 标签）。
- `.tsx`：含 JSX（也就是 `<div>` 这种类 HTML 写法）的文件，React 组件用它。

### 二、类型标注：在变量/参数后面用 `:` 写类型

```typescript
let name: string = 'George'   // name 只能是字符串
let age: number = 18          // 只能是数字
let ok: boolean = true        // 只能是 true / false
let list: string[] = ['a']    // 字符串数组（string[] 读作“string 的数组”）
```

常见基础类型：`string`、`number`、`boolean`、`null`、`undefined`、`void`（函数没有返回值）、`any`（放弃检查，任何类型，尽量少用）、`unknown`（未知类型，比 any 安全）。

### 三、函数的类型：参数类型 + 返回值类型

```typescript
// function 写法：参数 slug 是 string，返回值是 string
function loadPost(slug: string): string {
  return slug + '.mdx'
}

// 箭头函数写法（=> 是“箭头”，等价于一个匿名函数）
const add = (a: number, b: number): number => a + b
```

`async function`（异步函数）的返回值一定是一个 `Promise`（见第六点）。

### 四、`interface` 和 `type`：给“对象的形状”起名字

`interface` 用来描述一个对象应该长什么样，之后就能复用这个名字：

```typescript
interface Post {
  slug: string          // 必须有 slug，且是 string
  title: string
  readingTime?: number  // 末尾的 ? 表示“可选”，可有可无
}

// 用法：声明一个变量符合 Post 的形状
const p: Post = { slug: 'hello', title: 'Hi' }  // 不写 readingTime 也合法
```

`type` 作用类似，但更适合写“联合类型 / 别名”：

```typescript
type Locale = 'en' | 'zh'   // 联合类型：Locale 只能是 'en' 或 'zh' 这两个字符串之一
```

> `'en' | 'zh'` 这种“只能取某几个固定字符串”的写法叫**字面量联合类型**，是 TS 里非常常用的“枚举”手段。

### 五、可选 `?`、联合 `|`、`A & B`

- `字段?:` → 这个字段可有可无（可选属性）。
- `A | B` → 类型是 A **或** B（联合）。例：`string | null` 表示要么是字符串、要么是 null。
- `A & B` → 同时满足 A **和** B（交叉）。例：`Error & { digest?: string }` 表示“是个 Error，并且额外可能有 digest 字段”。

### 六、泛型 `<T>`：给类型“传参数”

泛型就是“类型里的占位符”，用尖括号 `<>` 传具体类型，最常见的几个：

```typescript
Promise<Post>          // 一个最终会得到 Post 的异步结果
Post[]                 // Post 数组，等价写法 Array<Post>
Record<string, string> // 一个对象，key 是 string、value 也是 string
Set<string>            // 元素都是 string 的集合
useState<Theme>(...)   // React 的状态，值类型被限定为 Theme
```

例：`function getPosts(): Promise<Post[]>` 表示“这是异步函数，最终拿到一个 Post 数组”。

### 七、类型断言 `as` 与 `as const`

- `值 as 类型`：告诉 TS “相信我，这个值就是这个类型”，用于 TS 推断不出来时。例：`JSON.parse(...) as DbJson`。
- `值 as unknown as 目标类型`：当两个类型差太远、不能直接 `as` 时的“强制两步转换”。
- `对象 as const`：把对象“锁死”成只读且取最窄的字面量类型，常用于翻译字典，让每个 key 都变成精确的字符串字面量。

### 八、`import type` 与 `.d.ts`

- `import type { Post } from '@/types'`：**只导入类型**（编译后这行会被完全删掉，不影响运行），用于纯类型用途，性能更好、语义更清楚。
- `xxx.d.ts`：**类型声明文件**，里面只写类型、不写实现，用来“告诉 TS 某些东西的类型”（例如告诉 TS：`.mdx` 文件能被 import）。
- `declare global { interface Window { ... } }`：往全局 `window` 对象上“补充声明”一个属性，否则 TS 会说“找不到这个属性”。

### 九、`keyof` 和 `typeof`（进阶，但本文用到了）

```typescript
const translations = { en: { 'nav.home': 'Home' } } as const
type Key = keyof typeof translations['en']
// typeof translations['en'] 取“en 这个对象的类型”
// keyof 再取“它所有的 key 组成的联合类型”，即 'nav.home' | ...
```

一句话：`typeof 变量` 拿到这个变量的类型；`keyof 类型` 拿到这个类型所有 key 的联合。

### 十、本文也大量用到的纯 JS 语法（不是 TS 特有）

| 写法 | 名字 | 作用 |
|------|------|------|
| `const { posts, total } = obj` | 解构 | 从对象里一次性取出多个字段 |
| `const [a, b] = arr` | 数组解构 | 按位置取出数组元素（`useState` 返回值就是这样取） |
| `function f({ page = 1 } = {})` | 默认值 | 不传参时用默认值；末尾 `= {}` 表示“整个参数对象也可以不传” |
| `...rest` / `{ ...obj }` | 剩余 / 展开 | 收集剩余项 / 把一个对象的字段“铺开”到另一个对象 |
| `a?.b` | 可选链 | 如果 a 是 null/undefined 就直接返回 undefined，不报错 |
| `a ?? b` | 空值合并 | a 是 null/undefined 时取 b（注意：0、'' 不算空，会保留） |
| `a \|\| b` | 逻辑或 | a 为“假值”(0/''/null/undefined/false) 时取 b |
| `cond ? x : y` | 三元表达式 | 条件成立取 x，否则取 y |
| `` `Hi ${name}` `` | 模板字符串 | 用反引号拼字符串，`${}` 里放变量 |
| `await fn()` | await | 等异步操作完成再继续（只能写在 async 函数里） |
| `arr.map(fn)` / `.filter` / `.sort` | 数组方法 | 遍历转换 / 过滤 / 排序，返回新数组 |

看完以上，下面的代码就基本没有“看不懂的符号”了。

### 第一天下午：类型定义文件

#### 2.1 创建 `types/index.ts` — 核心数据类型

> ⚠️ 这是与旧版差异最大的文件。类型必须与**真实组件签名**一致，否则迁移组件时会全是红线。

```typescript
import type { ReactNode } from 'react'

// ============ MDX 文章 ============
export interface PostFrontmatter {
  title: string
  description: string
  date: string
  tags?: string[]
  author?: string
}

export interface Post {
  slug: string
  frontmatter: PostFrontmatter
  // compileMDX 返回已编译好的 React 节点（详情页直接渲染），
  // 列表场景下可能为空。
  content: ReactNode
  readingTime?: number
}

// ============ 项目 / 作品 ============
// 既兼容 GitHub API 返回的字段，也兼容本地 db.json 的字段。
export interface Repo {
  id: number | string
  name: string
  title?: string // db.json fallback 使用的字段名
  description: string | null
  stargazers_count: number
  html_url?: string
  language?: string | null
}

// 迁移到 Prisma 后前端使用的项目类型
export interface Project {
  id: string
  name: string
  description: string | null
  url: string | null
  language: string | null
  stargazersCount: number
}

// ============ 分页 ============
// ⚠️ 与实际 components/pagination.tsx 对齐：组件只接收 pageCount，
// 当前页从 URL searchParams 中读取。
export interface PaginationProps {
  pageCount: number
}

// ============ i18n ============
export type Locale = 'en' | 'zh'

// ============ 主题 ============
export type Theme = 'dark' | 'light'

// ============ Cookie 状态 ============
export interface AppCookies {
  theme?: Theme
  lang?: Locale
}

// ============ 导航项 ============
export interface NavItem {
  href: string
  i18nKey: string
}
```

**📖 逐行解释**

- `import type { ReactNode } from 'react'`：从 react 里**只导入类型** `ReactNode`。`ReactNode` 是 React 里“任何能渲染的东西”的类型——字符串、数字、JSX 元素、它们的数组等都算。
- `export interface PostFrontmatter { ... }`：定义并导出一个“文章头信息”的形状。`title/description/date` 是必填字符串；`tags?: string[]` 和 `author?: string` 末尾有 `?`，表示**可选**（有的文章没有标签或作者）。
- `content: ReactNode`：文章正文。注意它不是 `string`，因为 MDX 被 `compileMDX` 编译后得到的是**可渲染的 React 节点**，所以用 `ReactNode`。
- `Repo` 里的 `id: number | string`：联合类型，GitHub API 给的是数字 id、本地数据可能是字符串，两种都允许。`description: string | null`：要么有描述、要么是 `null`。`html_url?` / `language?`：可选，因为本地 `db.json` 里没有这些字段。
- `export type Locale = 'en' | 'zh'`：字面量联合类型，全站语言只可能是这两个值之一。`Theme = 'dark' | 'light'` 同理。
- `PaginationProps { pageCount: number }`：分页组件**只需要**总页数这一个 prop（当前页它自己从网址读），所以类型里只有一个字段——这正是修正旧版错误的关键。
- 整体作用：这个文件是“类型中心”，其它文件用 `import type { Post } from '@/types'` 复用这些定义，保证全站对同一份数据的理解一致。

#### 2.2 创建 `types/mdx.d.ts` — MDX 模块声明

```typescript
declare module '*.mdx' {
  import type { MDXProps } from 'mdx/types'
  // @types/react 19 移除了全局 JSX 命名空间，统一使用 React.JSX
  const MDXComponent: (props: MDXProps) => React.JSX.Element
  export default MDXComponent
  export const frontmatter: import('./index').PostFrontmatter
}
```

**📖 逐行解释**

- 这是个 `.d.ts`（类型声明文件），里面**只声明类型、不写实现**。
- `declare module '*.mdx' { ... }`：告诉 TS——“凡是 import 一个 `.mdx` 文件，它就是下面声明的这种模块”。否则 `import Post from './a.mdx'` 时 TS 会报“找不到模块”。`*` 是通配符，匹配所有 `.mdx`。
- `const MDXComponent: (props: MDXProps) => React.JSX.Element`：声明默认导出是一个**组件函数**——接收 `MDXProps` 参数、返回一个 JSX 元素。这里用 `React.JSX.Element` 而不是 `JSX.Element`，因为 `@types/react@19` 删掉了全局 `JSX`。
- `export const frontmatter: import('./index').PostFrontmatter`：声明每个 mdx 还会导出一个 `frontmatter`，类型复用我们在 `index.ts` 里定义的 `PostFrontmatter`。`import('./index').X` 是“在类型位置内联引用另一个文件里类型”的写法。

#### 2.3 创建 `types/env.d.ts` — 环境变量声明（为 Supabase 预留）

```typescript
declare namespace NodeJS {

    NEXT_PUBLIC_SUPABASE_ANON_KEY?: string
    DATABASE_URL?: string
    GITHUB_TOKEN?: string

```

**📖 逐行解释**

- 同样是 `.d.ts` 声明文件。`declare namespace NodeJS { interface ProcessEnv { ... } }` 是在“给已有的 Node 类型打补丁”——往 `process.env` 的类型里补充我们自己的环境变量名。
- 这样写之后，你在代码里写 `process.env.DATABASE_URL` 会有自动补全，且类型是 `string | undefined`（因为加了 `?`，表示运行时可能没配置）。

### 第二天：核心文件逐批迁移

> 迁移策略：**从底层到上层，从无依赖到有依赖**，确保每批次迁移后项目仍可运行。

#### 第一批：工具库（lib/）

##### `lib/posts.ts`（文件系统版本，TS 迁移阶段保持现有 MDX 逻辑）

> ⚠️ `.ts` 文件不能写 JSX，`compileMDX` 的 `components` 里用 `React.createElement` 代替 `<H1 />`。

```typescript
import fs from 'fs'
import path from 'path'
import React from 'react'
import { compileMDX } from 'next-mdx-remote/rsc'
import H1 from '@/components/h1'
import type { Post } from '@/types'

export function loadPost(slug: string): string {
  const filename = slug.endsWith('.mdx') ? slug : `${slug}.mdx`
  return fs.readFileSync(
    path.join(process.cwd(), 'content', filename),
    'utf-8'
  )
}

export async function getPostBySlug(slug: string): Promise<Post> {
  const source = loadPost(slug)

  const { frontmatter, content } = await compileMDX<Post['frontmatter']>({
    source,
    components: {
      h1: (props) => React.createElement(H1, props),
    },
    options: {
      parseFrontmatter: true,
    },
  })

  return {
    slug: slug.replace('.mdx', ''),
    frontmatter,
    content,
  }
}

export async function getPosts({
  newest = true,
  page = 1,
  limit = 3,
  tags,
}: {
  newest?: boolean
  page?: number
  limit?: number
  tags?: string[]
} = {}): Promise<{ posts: Post[]; total: number; totalPages: number }> {
  const files = fs.readdirSync(path.join(process.cwd(), 'content'))

  const allPosts = await Promise.all(
    files.map((filename) => getPostBySlug(filename))
  )

  let filteredPosts = allPosts

  if (tags && tags.length > 0) {
    filteredPosts = filteredPosts.filter((post) =>
      post.frontmatter.tags?.some((tag) => tags.includes(tag))
    )
  }

  filteredPosts.sort((a, b) => {
    const dateA = new Date(a.frontmatter.date).getTime()
    const dateB = new Date(b.frontmatter.date).getTime()
    return newest ? dateB - dateA : dateA - dateB
  })

  const startIndex = (page - 1) * limit
  const endIndex = page * limit

  return {
    posts: filteredPosts.slice(startIndex, endIndex),
    total: filteredPosts.length,
    totalPages: Math.ceil(filteredPosts.length / limit),
  }
}

export async function getAllTags(): Promise<string[]> {
  const { posts } = await getPosts({ limit: 1000 })
  const tags = new Set<string>()
  posts.forEach((post) => {
    post.frontmatter.tags?.forEach((tag) => tags.add(tag))
  })
  return Array.from(tags).sort()
}
```

**📖 逐行解释**

- `import type { Post } from '@/types'`：只导入 `Post` 类型（不参与运行）。`@/` 是 `tsconfig.json` 里配的路径别名，等于项目根目录，避免写一长串 `../../`。
- `loadPost(slug: string): string`：参数 `slug` 标了 `string`，返回值标了 `string`。函数体读取 `content/xxx.mdx` 文件并返回文件内容字符串。`slug.endsWith('.mdx') ? slug : `${slug}.mdx``：三元 + 模板字符串，slug 没带 `.mdx` 后缀就补上。
- `getPostBySlug(slug: string): Promise<Post>`：因为是 `async`，返回值写成 `Promise<Post>`（“最终会得到一个 Post”）。
- `compileMDX<Post['frontmatter']>({...})`：`compileMDX` 是泛型函数，尖括号里传入 frontmatter 的类型。`Post['frontmatter']` 是“**索引访问类型**”，意思是“取 Post 这个类型里 frontmatter 字段的类型”，也就是 `PostFrontmatter`。这样编译出来的 `frontmatter` 变量就有了正确类型，能安全地 `.title`、`.tags`。
- `const { frontmatter, content } = await compileMDX(...)`：先 `await` 等编译完成，再用**解构**一次取出两个字段。
- `h1: (props) => React.createElement(H1, props)`：⚠️ 关键点——`.ts` 文件**不能写 JSX**（`<H1 />` 会被当成类型语法报错），所以用 `React.createElement(H1, props)` 这种纯函数写法达到同样效果。
- `getPosts({ newest = true, page = 1, ... }: { ... } = {})`：这是“带类型的解构参数 + 默认值”。冒号后面那一大块 `{ newest?: boolean; ... }` 是**参数的类型**（每个都加了 `?` 表示可不传）；最后的 `= {}` 表示“连整个参数对象都可以不传”，这样 `getPosts()` 不带参数也能调用。
- 返回类型 `Promise<{ posts: Post[]; total: number; totalPages: number }>`：直接把返回对象的形状内联写出来——一个含 `posts`（Post 数组）、`total`、`totalPages` 的对象。
- `filteredPosts.filter((post) => post.frontmatter.tags?.some((tag) => tags.includes(tag)))`：纯 JS 数组方法。`?.` 是可选链——文章可能没有 tags，避免对 `undefined` 调用 `.some` 报错。
- `new Set<string>()`：泛型——声明一个“只装字符串”的集合（Set 自动去重）。`Array.from(tags).sort()` 把 Set 转回数组再排序。
- 这一版仍然**从文件系统读 MDX**，只是补齐了类型；到 Day 4-5 才会换成从数据库读。

##### `hooks/use-server-language.ts`

```typescript
import { cookies, headers } from 'next/headers'
import type { Locale } from '@/types'

const useServerLanguage = (): Locale => {
  try {
    const cookieStore = cookies()
    const langCookie = cookieStore.get('lang')

    // 用户显式选择过语言 → 尊重其选择
    if (langCookie?.value === 'en' || langCookie?.value === 'zh') {
      return langCookie.value
    }

    // 无 cookie → 按 IP 国家判断
    const headersList = headers()
    const country = headersList.get('x-vercel-ip-country')

    if (country === 'CN') {
      return 'zh'
    }

    return 'en'
  } catch {
    return 'en'
  }
}

export default useServerLanguage
```

**📖 逐行解释**

- `(): Locale =>`：这个 hook 不收参数，返回值类型是 `Locale`（`'en' | 'zh'`）。函数里所有 `return` 都只返回 `'en'` 或 `'zh'`，正好符合。
- `cookies()` / `headers()` 来自 `next/headers`，是 Next.js 服务端读取 Cookie 和请求头的 API，类型由 Next 自动提供，不用我们标注。
- `if (langCookie?.value === 'en' || langCookie?.value === 'zh')`：`?.` 可选链——cookie 可能不存在；只有值确实是 `'en'`/`'zh'` 时才返回它。这里**特意做了值校验**，因为 cookie 取出来本是任意字符串，校验后 TS 才允许把它当作 `Locale` 返回。
- `catch { return 'en' }`：出错（比如不在请求上下文里）就兜底英文。`catch` 后不写 `(e)` 是新语法“可选 catch 绑定”，表示不关心错误对象。

##### `hooks/use-dark-mode.ts`

```typescript
import { useState, useEffect } from 'react'
import { useCookies } from 'react-cookie'
import type { Theme } from '@/types'

const useDarkMode = (defaultTheme: Theme = 'dark') => {
  const [theme, setTheme] = useState<Theme>(defaultTheme)
  const [, setCookie] = useCookies(['theme'])
  const [mounted, setMounted] = useState(false)

  useEffect(() => {
    setMounted(true)
  }, [])

  const setAndSaveTheme = (newTheme: Theme) => {
    setTheme(newTheme)
    document.documentElement.classList.remove('light', 'dark')
    document.documentElement.classList.add(newTheme)
    setCookie('theme', newTheme, { path: '/' })
  }

  const toggleTheme = () => {
    setAndSaveTheme(theme === 'dark' ? 'light' : 'dark')
  }

  return { theme, toggleTheme, mounted }
}

export default useDarkMode
```

**📖 逐行解释**

- `useState<Theme>(defaultTheme)`：`useState` 是 React 的状态钩子，返回 `[当前值, 修改函数]`。这里用泛型 `<Theme>` 把状态值限定为 `'dark' | 'light'`，于是 `setTheme('blue')` 会被 TS 拦下。
- `const [theme, setTheme] = useState<Theme>(...)`：**数组解构**——按位置取出“当前值 theme”和“修改函数 setTheme”。
- `const [, setCookie] = useCookies(['theme'])`：第一个位置留空（逗号前什么都不写）表示“我不需要读 cookie 值，只要写入函数”。
- `const [mounted, setMounted] = useState(false)`：没写泛型，TS 会**自动推断**为 `boolean`（因为初始值是 `false`），这种能推断出来的就不用手动标。
- `setAndSaveTheme = (newTheme: Theme) => {...}`：参数标 `Theme`。函数体操作 DOM 类名并写入 cookie，是普通 JS DOM 操作。
- 返回 `{ theme, toggleTheme, mounted }`：把状态和方法打包成对象返回，组件里再解构使用。

##### `hooks/use-language.ts`

```typescript
'use client'

import { useState, useEffect } from 'react'
import { useRouter } from 'next/navigation'
import { useCookies } from 'react-cookie'
import { getTranslation } from '@/lib/i18n'
import type { Locale } from '@/types'

const useLanguage = (defaultLang: Locale = 'en') => {
  const router = useRouter()
  const [cookies, setCookie] = useCookies(['lang'])
  const initialLang: Locale = (cookies.lang as Locale) || defaultLang
  const [lang, setLang] = useState<Locale>(initialLang)
  const [mounted, setMounted] = useState(false)

  useEffect(() => {
    if (cookies.lang && cookies.lang !== lang) {
      setLang(cookies.lang as Locale)
    }
    setMounted(true)
  }, [cookies.lang, lang])

  const setAndSaveLang = (newLang: Locale) => {
    setLang(newLang)
    document.documentElement.lang = newLang
    setCookie('lang', newLang, { path: '/' })
  }

  const toggleLang = () => {
    const newLang: Locale = lang === 'en' ? 'zh' : 'en'
    setAndSaveLang(newLang)
    router.refresh()
  }

  const t = (key: string): string => {
    const dict = getTranslation(lang) as Record<string, string>
    return dict[key] || key
  }

  return { lang, toggleLang, t, mounted }
}

export default useLanguage
```

**📖 逐行解释**

- `'use client'`：放在文件最顶部，告诉 Next.js“这是客户端组件”（能用 `useState`、浏览器 API、事件等）。带交互的 hook/组件都要加。
- `const initialLang: Locale = (cookies.lang as Locale) || defaultLang`：`cookies.lang` 从 cookie 取出来类型是宽泛的 `string`，用 `as Locale` 断言成我们要的窄类型；`||` 在它为空时退回默认值。
- `useState<Locale>(initialLang)`：状态值限定为 `Locale`。
- `setLang(cookies.lang as Locale)`：同样需要 `as Locale`，因为来源是 string。
- `setAndSaveLang = (newLang: Locale) => {...}` / `const newLang: Locale = lang === 'en' ? 'zh' : 'en'`：参数和变量都标 `Locale`；三元表达式实现“en↔zh 切换”。
- `const t = (key: string): string => {...}`：翻译函数——参数 key 是 string，返回 string。`getTranslation(lang) as Record<string, string>` 把字典断言成“任意字符串 key 都能取”，于是 `dict[key]` 不报错；`dict[key] || key` 表示找不到翻译就原样显示 key（兜底，避免页面空白）。

'use client'

  const [count, setCount] = useState(0)

**📖 逐行解释**

> ⚠️ Props 是 `{ pageCount }`，当前页从 `searchParams` 读取。

```typescript
'use client'

import Link from 'next/link'
import { useSearchParams, usePathname } from 'next/navigation'
import { motion } from 'framer-motion'
import type { PaginationProps } from '@/types'

export default function Pagination({ pageCount }: PaginationProps) {
  const searchParams = useSearchParams()
  const pathname = usePathname()

  if (pageCount <= 1) return null

  const currentPage = Number(searchParams.get('page') ?? 1)
  const pages: number[] = []
  for (let i = 1; i <= pageCount; i++) {
    pages.push(i)
  }

  return (
    <nav className="flex justify-center items-center gap-2">
      {pages.map((pageNumber) => {
        const params = new URLSearchParams(searchParams)
        params.set('page', pageNumber.toString())
        const isActive = pageNumber === currentPage

        return (
          <Link
            key={pageNumber}
            href={`${pathname}?${params.toString()}`}
            className="relative"
          >
            <motion.span
              className={`inline-flex items-center justify-center w-10 h-10 rounded-xl text-sm font-mono font-medium transition-colors duration-300 ${
                isActive
                  ? 'text-white bg-[var(--accent)]'
                  : 'text-[var(--text-secondary)] hover:text-[var(--accent)] hover:bg-[var(--bg-secondary)]'
              }`}
              whileHover={!isActive ? { scale: 1.1 } : {}}
              whileTap={!isActive ? { scale: 0.95 } : {}}
            >
              {pageNumber}
            </motion.span>
          </Link>
        )
      })}
    </nav>
  )
}
```

**📖 逐行解释**

- `export default function Pagination({ pageCount }: PaginationProps)`：从 props 里解构出 `pageCount`，类型用我们定义的 `PaginationProps`（只有这一个字段）。这是修正旧版“假设有 currentPage/basePath”的关键。
- `if (pageCount <= 1) return null`：只有一页时不显示分页。组件返回 `null` 表示“什么都不渲染”，这在 React 里完全合法。
- `const currentPage = Number(searchParams.get('page') ?? 1)`：**当前页不是 prop，而是从网址查询参数里读**（`?page=2`）。`searchParams.get('page')` 取不到时用 `?? 1` 兜底，`Number(...)` 把字符串转成数字。
- `const pages: number[] = []`：声明一个 number 数组，用 `for` 循环 push 进 1..pageCount。
- `params.set('page', pageNumber.toString())`：给每个页码生成对应链接的查询参数；`.toString()` 把 number 转回 string（URLSearchParams 只接受字符串）。

##### `components/page-transition.tsx`

```typescript
'use client'

import { motion, AnimatePresence, type Variants } from 'framer-motion'
import { usePathname } from 'next/navigation'
import type { ReactNode } from 'react'

const pageVariants: Variants = {
  initial: { opacity: 0, y: 24, scale: 0.98, filter: 'blur(4px)' },
  animate: {
    opacity: 1,
    y: 0,
    scale: 1,
    filter: 'blur(0px)',
    transition: { duration: 0.6, ease: [0.4, 0, 0.2, 1], staggerChildren: 0.08 },
  },
  exit: {
    opacity: 0,
    y: -12,
    filter: 'blur(4px)',
    transition: { duration: 0.35, ease: [0.4, 0, 0.2, 1] },
  },
}

const childVariants: Variants = {
  initial: { opacity: 0, y: 16 },
  animate: {
    opacity: 1,
    y: 0,
    transition: { duration: 0.5, ease: [0.4, 0, 0.2, 1] },
  },
}

interface PageTransitionProps {
  children: ReactNode
  className?: string
}

export default function PageTransition({
  children,
  className = '',
}: PageTransitionProps) {
  const pathname = usePathname()

  return (
    <AnimatePresence mode="wait">
      <motion.div
        key={pathname}
        variants={pageVariants}
        initial="initial"
        animate="animate"
        exit="exit"
        className={className}
      >
        {children}
      </motion.div>
    </AnimatePresence>
  )
}

interface MotionItemProps {
  children: ReactNode
  className?: string
  delay?: number
}

export function MotionItem({
  children,
  className = '',
  delay = 0,
}: MotionItemProps) {
  return (
    <motion.div
      variants={childVariants}
      initial="initial"
      animate="animate"
      transition={{ delay }}
      className={className}
    >
      {children}
    </motion.div>
  )
}
```

**📖 逐行解释**

- `import { motion, AnimatePresence, type Variants } from 'framer-motion'`：注意 `type Variants`——在**同一行 import 里**用 `type` 关键字标记“这一个是类型导入”，其余 `motion`/`AnimatePresence` 是值导入。这是混合导入的常见写法。
- `const pageVariants: Variants = {...}`：动画配置对象，标成 framer-motion 的 `Variants` 类型，这样写错动画字段名会被提示。`childVariants` 同理。
- 这个文件导出**两个**东西：`export default function PageTransition`（默认导出）和 `export function MotionItem`（命名导出）。所以别处用 `import PageTransition, { MotionItem } from ...`。
- 两个组件都定义了自己的 props interface（`PageTransitionProps` / `MotionItemProps`），`children: ReactNode`、`className?`、`delay?: number` 都是前面见过的套路。

##### `components/chatbot.tsx`

> ⚠️ `initializeChatbot` 是 `public/chatbot.js` 注入到 `window` 的全局函数，TS 下需要声明全局类型，否则报 “Cannot find name 'initializeChatbot'”。

```typescript
'use client'

import Script from 'next/script'

declare global {
  interface Window {
    initializeChatbot?: () => void
  }
}

export default function Chatbot() {
  return (
    <Script
      src="/chatbot.js"
      strategy="lazyOnload"
      onLoad={() => window.initializeChatbot?.()}
    />
  )
}
```

**📖 逐行解释**

- `declare global { interface Window { initializeChatbot?: () => void } }`：⚠️ 关键点。`initializeChatbot` 是外部脚本 `public/chatbot.js` 挂到浏览器全局 `window` 上的函数，TS 默认不知道它存在，直接调用会报“找不到名称”。这段 `declare global` 就是“补充声明”——告诉 TS：window 上可能有个名叫 `initializeChatbot` 的可选函数，它没有返回值（`() => void`）。
- `onLoad={() => window.initializeChatbot?.()}`：脚本加载完后调用它。`?.()` 是“可选调用”——只有该函数真的存在时才执行，否则跳过，避免报错。

> 注意：`public/chatbot.js` 是**静态资源**，由浏览器直接加载，**不要**迁移成 TS。

- 其余 `<H1>`、`<MotionItem>`、`<Card>`、Tailwind 类名都是 UI，逻辑同 JS 版。

- `let repos: Repo[] = []` / `let error: string | null = null`：显式标类型的可变变量。`error` 用联合类型 `string | null`——初始是 null，出错时才赋字符串。
- `const sortedRepos = [...repos].sort(...)`：`[...repos]` 先**复制一份**再排序（避免直接改原数组），按 star 数降序。
- `{repo.name ?? repo.title}`：`??` 兜底——GitHub 数据有 `name`，本地 db.json 只有 `title`，两边兼容。

#### 4.3 设计数据库 Schema

创建/覆盖 `prisma/schema.prisma`：

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}

// ============ 文章模型 ============
model Post {
  id          String    @id @default(cuid())
  slug        String    @unique
  title       String
  description String
  content     String                      // MDX 源码
  date        DateTime
  author      String?
  published   Boolean   @default(true)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  tags        PostTag[]
}

// ============ 标签模型 ============
model Tag {
  id    String    @id @default(cuid())
  name  String    @unique
  slug  String    @unique
  posts PostTag[]
}

// ============ 文章-标签 多对多关联 ============
model PostTag {
  postId String
  tagId  String
  post   Post @relation(fields: [postId], references: [id], onDelete: Cascade)
  tag    Tag  @relation(fields: [tagId], references: [id], onDelete: Cascade)

  @@id([postId, tagId])
}

// ============ 项目 / 作品模型 ============
model Project {
  id              String   @id @default(cuid())
  name            String
  description     String?
  url             String?
  language        String?
  stargazersCount Int      @default(0)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
}

// ============ 用户模型（预留，后续接入认证） ============
model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

**📖 逐行解释（这是 Prisma 自己的语法，不是 TS）**

- `generator client {...}`：告诉 Prisma 生成一个 JS/TS 客户端（之后 `import { PrismaClient } from '@prisma/client'` 用的就是它）。
- `datasource db { provider = "postgresql"; url = env("DATABASE_URL"); directUrl = env("DIRECT_URL") }`：数据库连接配置。`env("X")` 表示从环境变量读取。`url` 给应用日常用（连接池），`directUrl` 给迁移用（直连）。
- `model Post {...}`：定义一张表。每行是一个字段：`字段名 类型 修饰符`。
  - `id String @id @default(cuid())`：主键，默认生成一个随机字符串 id。
  - `slug String @unique`：唯一字段（不能重复）。
  - `content String`：存 MDX 源码。`date DateTime`：日期。
  - `author String?`：字段类型后的 `?` 表示**可空**（这点和 TS 的可选含义相通）。
  - `published Boolean @default(true)`：默认值 true。`createdAt @default(now())` / `updatedAt @updatedAt`：自动维护的创建/更新时间。
  - `tags PostTag[]`：关系字段——一篇文章关联多条 `PostTag`。
- `model PostTag {...}`：**多对多中间表**。文章和标签是多对多（一篇文多个标签、一个标签多篇文），数据库要用一张中间表连接。`@relation(fields: [postId], references: [id], onDelete: Cascade)` 指明外键关系，`onDelete: Cascade` 表示文章删了关联也跟着删。`@@id([postId, tagId])` 用两个字段做联合主键。
- `model Project` / `model User`：项目表和预留的用户表，写法同理。
- 写完这个 schema，下一步 `prisma migrate` 会据此**自动在数据库里建表**，并生成带完整 TS 类型的客户端。

#### 4.5 创建 Prisma Client 单例

创建 `lib/prisma.ts`：

```typescript
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined
}

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    log: process.env.NODE_ENV === 'development' ? ['error', 'warn'] : ['error'],
  })

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma
}
```

**📖 逐行解释**

- 这段是 Prisma 的“**单例模式**”标准写法，目的：开发时 Next 热重载会反复执行模块，若每次都 `new PrismaClient()` 会创建大量数据库连接、最终报错。所以把实例挂到全局，复用同一个。
- `const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined }`：`globalThis` 是全局对象，但 TS 默认不知道它上面有 `prisma`。`as unknown as {...}` 是**两步强制断言**——先转成 `unknown`（万能中转）、再转成我们想要的形状，从而能安全访问 `globalForPrisma.prisma`。
- `export const prisma = globalForPrisma.prisma ?? new PrismaClient({...})`：用 `??` ——全局已有就复用，没有才新建。
- `log: process.env.NODE_ENV === 'development' ? ['error', 'warn'] : ['error']`：三元——开发环境多打点日志。
- `if (process.env.NODE_ENV !== 'production') { globalForPrisma.prisma = prisma }`：只在非生产环境把实例存回全局（生产环境不需要，因为不会热重载）。

#### 5.1 重写 `lib/posts.ts` — 从文件系统改为 Prisma

> 关键：列表只取 frontmatter 字段，**不编译 MDX**（性能）；详情页才取 `content` 源码并 `compileMDX`。

```typescript
import React from 'react'
import { compileMDX } from 'next-mdx-remote/rsc'
import { prisma } from './prisma'
import H1 from '@/components/h1'
import type { Post } from '@/types'
import type { Prisma } from '@prisma/client'

// 带标签关联的 Prisma 文章类型
type PostWithTags = Prisma.PostGetPayload<{
  include: { tags: { include: { tag: true } } }
}>

// Prisma 记录 → 前端使用的 frontmatter（不含编译内容）
function toFrontmatter(record: PostWithTags) {
  return {
    title: record.title,
    description: record.description,
    date: record.date.toISOString().split('T')[0],
    tags: record.tags.map((pt) => pt.tag.name),
    author: record.author ?? undefined,
  }
}

export async function getPosts({
  newest = true,
  page = 1,
  limit = 3,
  tags,
}: {
  newest?: boolean
  page?: number
  limit?: number
  tags?: string[]
} = {}): Promise<{ posts: Post[]; total: number; totalPages: number }> {
  const where: Prisma.PostWhereInput = {
    published: true,
    ...(tags && tags.length > 0
      ? { tags: { some: { tag: { name: { in: tags } } } } }
      : {}),
  }

  const [records, total] = await Promise.all([
    prisma.post.findMany({
      where,
      include: { tags: { include: { tag: true } } },
      orderBy: { date: newest ? 'desc' : 'asc' },
      skip: (page - 1) * limit,
      take: limit,
    }),
    prisma.post.count({ where }),
  ])

  return {
    posts: records.map((record) => ({
      slug: record.slug,
      frontmatter: toFrontmatter(record),
      content: null, // 列表不需要正文
    })),
    total,
    totalPages: Math.ceil(total / limit),
  }
}

export async function getPostBySlug(slug: string): Promise<Post | null> {
  const record = await prisma.post.findUnique({
    where: { slug },
    include: { tags: { include: { tag: true } } },
  })

  if (!record) return null

  // 把存储的 MDX 源码编译成可渲染的 React 节点
  const { content } = await compileMDX({
    source: record.content,
    components: {
      h1: (props) => React.createElement(H1, props),
    },
    options: { parseFrontmatter: false },
  })

  return {
    slug: record.slug,
    frontmatter: toFrontmatter(record),
    content,
  }
}

export async function getAllTags(): Promise<string[]> {
  const tags = await prisma.tag.findMany({ orderBy: { name: 'asc' } })
  return tags.map((t) => t.name)
}
```

**📖 逐行解释**

- `import type { Prisma } from '@prisma/client'`：导入 Prisma **自动生成**的类型集合（不用手写、也不用装 `@types/prisma`）。
- `type PostWithTags = Prisma.PostGetPayload<{ include: { tags: { include: { tag: true } } } }>`：这是 Prisma 提供的“**根据查询形状推导结果类型**”的泛型工具。因为我们查询时 `include` 了关联的 tags，返回的对象会比裸 Post 多出 `tags` 字段；`PostGetPayload<...>` 就能精确算出“带 tags 的 Post”长什么样，省去手写。
- `function toFrontmatter(record: PostWithTags) {...}`：把数据库记录转成前端要的 frontmatter 形状。`record.date.toISOString().split('T')[0]` 把日期对象转成 `'2024-01-19'` 这种字符串；`record.tags.map((pt) => pt.tag.name)` 从中间表里把标签名取出来；`record.author ?? undefined` 把数据库的 `null` 统一成 `undefined`（贴合我们 `author?: string` 的定义）。
- `getPosts(...)` 列表查询：`const where: Prisma.PostWhereInput = {...}` 用 Prisma 的“查询条件类型”。里面 `...(tags && tags.length > 0 ? {...} : {})` 是**条件展开**——有标签筛选就把筛选条件铺进对象，没有就铺一个空对象（相当于不加该条件）。
- `const [records, total] = await Promise.all([findMany(...), count(...)])`：`Promise.all` 让“查列表”和“查总数”**并行执行**再一起拿结果，比一个个 await 快。数组解构取出两个结果。
- 列表项里 `content: null`：列表不需要正文，所以不编译 MDX、直接给 null（省性能）。
- `getPostBySlug(...): Promise<Post | null>`：返回值是 `Post | null`——查不到文章就返回 `null`。`if (!record) return null` 提前返回。查到后才 `compileMDX` 把存的 MDX 源码编译成可渲染节点（这一步不能省，否则页面显示的是一堆源码字符串）。

> ⚠️ `getPostBySlug` 现在返回 `Post | null`。详情页 `app/blog/[slug]/page.tsx` 需相应调整：

```typescript
export default async function BlogPage({ params }: BlogPostPageProps) {
  const post = await getPostBySlug(params.slug)
  if (!post) {
    notFound()
  }
  // ...其余渲染逻辑同上（用 post.frontmatter / post.content）
}
```

同样 `generateMetadata` 改成：

```typescript
export async function generateMetadata({
  params,
}: BlogPostPageProps): Promise<Metadata> {
  const post = await getPostBySlug(params.slug)
  if (!post) return {}
  return {
    title: post.frontmatter.title,
    description: post.frontmatter.description || '',
  }
}
```

**📖 为什么要这样改**

- 文件系统版的 `getPostBySlug` 找不到会抛异常（用 `try/catch` 接），而 Prisma 版改成“找不到返回 `null`”。所以详情页要从 `try/catch` 改成 `const post = await getPostBySlug(...); if (!post) notFound()`。
- 关键点：TS 知道 `getPostBySlug` 返回 `Post | null`，如果你不先 `if (!post)` 处理掉 null 就直接 `post.frontmatter`，TS 会报错“对象可能为 null”——这正是类型在帮你堵住空指针 Bug。

#### 5.3 创建种子脚本 — 把现有内容迁进数据库

创建 `prisma/seed.ts`（完整实现，无占位）：

```typescript
import { PrismaClient } from '@prisma/client'
import fs from 'fs'
import path from 'path'
import matter from 'gray-matter'

const prisma = new PrismaClient()

function slugify(value: string): string {
  return value.toLowerCase().trim().replace(/\s+/g, '-')
}

interface DbJson {
  repos: {
    id: number
    title: string
    description: string | null
    stargazers_count: number
  }[]
}

async function seedPostsAndTags() {
  const contentDir = path.join(process.cwd(), 'content')
  const files = fs.readdirSync(contentDir).filter((f) => f.endsWith('.mdx'))

  for (const file of files) {
    const raw = fs.readFileSync(path.join(contentDir, file), 'utf-8')
    const { data, content } = matter(raw)

    const slug = file.replace(/\.mdx$/, '')
    const tagNames: string[] = Array.isArray(data.tags) ? data.tags : []

    // 先 upsert 文章本身
    const post = await prisma.post.upsert({
      where: { slug },
      update: {
        title: data.title ?? slug,
        description: data.description ?? '',
        content,
        date: data.date ? new Date(data.date) : new Date(),
        author: data.author ?? null,
        published: true,
      },
      create: {
        slug,
        title: data.title ?? slug,
        description: data.description ?? '',
        content,
        date: data.date ? new Date(data.date) : new Date(),
        author: data.author ?? null,
        published: true,
      },
    })

    // upsert 标签并建立关联
    for (const name of tagNames) {
      const tag = await prisma.tag.upsert({
        where: { name },
        update: {},
        create: { name, slug: slugify(name) },
      })

      await prisma.postTag.upsert({
        where: { postId_tagId: { postId: post.id, tagId: tag.id } },
        update: {},
        create: { postId: post.id, tagId: tag.id },
      })
    }

    console.log(`✔ seeded post: ${slug} (${tagNames.length} tags)`)
  }
}

async function seedProjects() {
  const dbJsonPath = path.join(process.cwd(), 'db.json')
  if (!fs.existsSync(dbJsonPath)) {
    console.log('⚠ db.json 不存在，跳过项目种子')
    return
  }

  const dbJson = JSON.parse(fs.readFileSync(dbJsonPath, 'utf-8')) as DbJson

  for (const repo of dbJson.repos) {
    await prisma.project.upsert({
      where: { id: String(repo.id) },
      update: {
        name: repo.title,
        description: repo.description,
        stargazersCount: repo.stargazers_count,
      },
      create: {
        id: String(repo.id),
        name: repo.title,
        description: repo.description,
        stargazersCount: repo.stargazers_count,
      },
    })
    console.log(`✔ seeded project: ${repo.title}`)
  }
}

async function main() {
  await seedPostsAndTags()
  await seedProjects()
  console.log('✅ Seed completed!')
}

main()
  .catch((e) => {
    console.error(e)
    process.exit(1)
  })
  .finally(async () => {
    await prisma.$disconnect()
  })
```

**📖 逐行解释**

- 这个脚本只在你本机跑一次，作用是把 `content/` 里的 MDX 和 `db.json` 里的项目“灌进”数据库。
- `import matter from 'gray-matter'`：`gray-matter` 库用来解析 MDX 顶部的 frontmatter。`const { data, content } = matter(raw)`：`data` 是解析出的头信息对象（title/date/tags…），`content` 是去掉头信息后的正文。
- `function slugify(value: string): string`：把标签名转成 url 友好的 slug（小写、空格变 `-`）。
- `interface DbJson { repos: {...}[] }`：给 `db.json` 的结构定义类型，这样 `JSON.parse(...) as DbJson` 之后能安全访问 `dbJson.repos`。
- `const tagNames: string[] = Array.isArray(data.tags) ? data.tags : []`：`data` 来自解析，类型很宽泛，先用 `Array.isArray` 判断 tags 真是数组再用，否则给空数组兜底。
- `prisma.post.upsert({ where, update, create })`：**upsert = update or insert**——按 `where`（这里是 slug）找：找到就用 `update` 更新、没找到就用 `create` 新建。好处是这个脚本可以**重复运行**而不会插入重复数据。
- 标签关联：先 `prisma.tag.upsert` 确保标签存在，再 `prisma.postTag.upsert` 在中间表建立“文章↔标签”这条连线。`where: { postId_tagId: { postId, tagId } }` 用的是 schema 里那个联合主键的查询写法。
- 结尾的 `main().catch(...).finally(...)`：跑主流程，出错就打印并 `process.exit(1)`（非零退出码=失败），无论成败最后都 `prisma.$disconnect()` 断开连接。

在 `package.json` 顶层添加 `prisma.seed` 配置：

```json
{
  "prisma": {
    "seed": "tsx prisma/seed.ts"
  }
}
```

运行：

```bash
npx prisma db seed
```

#### 6.1 创建 API Route（展示 Route Handler 能力）

创建 `app/api/posts/route.ts`：

```typescript
import { NextResponse, type NextRequest } from 'next/server'
import { prisma } from '@/lib/prisma'
import type { Prisma } from '@prisma/client'

export async function GET(request: NextRequest) {
  const searchParams = request.nextUrl.searchParams
  const page = parseInt(searchParams.get('page') ?? '1', 10)
  const limit = parseInt(searchParams.get('limit') ?? '10', 10)
  const tag = searchParams.get('tag')

  const where: Prisma.PostWhereInput = {
    published: true,
    ...(tag ? { tags: { some: { tag: { slug: tag } } } } : {}),
  }

  const [posts, total] = await Promise.all([
    prisma.post.findMany({
      where,
      include: { tags: { include: { tag: true } } },
      orderBy: { date: 'desc' },
      skip: (page - 1) * limit,
      take: limit,
    }),
    prisma.post.count({ where }),
  ])

  return NextResponse.json({
    posts: posts.map((p) => ({
      id: p.id,
      slug: p.slug,
      title: p.title,
      description: p.description,
      date: p.date,
      tags: p.tags.map((pt) => pt.tag.name),
    })),
    total,
    page,
    totalPages: Math.ceil(total / limit),
  })
}
```

**📖 逐行解释**

- 这是一个 **API 接口**（Route Handler）。文件名固定为 `route.ts`，导出名为 `GET` 的函数就会响应对 `/api/posts` 的 GET 请求。
- `import { NextResponse, type NextRequest } from 'next/server'`：同行混合导入——`NextResponse` 是值（要用它返回 JSON），`type NextRequest` 是类型（只用于标注参数）。
- `export async function GET(request: NextRequest)`：参数 `request` 标成 `NextRequest`，于是 `request.nextUrl.searchParams` 有类型提示。
- `parseInt(searchParams.get('page') ?? '1', 10)`：从 url 取 `page`，取不到用 `'1'` 兜底，`parseInt(_, 10)` 按十进制转成数字。
- `const where: Prisma.PostWhereInput = { published: true, ...(tag ? {...} : {}) }`：和数据层一样的条件展开写法——传了 tag 才加标签筛选。
- `return NextResponse.json({...})`：把查询结果包成 JSON 返回。`posts.map((p) => ({ id: p.id, ... }))` 只挑前端需要的字段返回（不暴露多余数据）。注意 `=> ({...})` 又是“箭头函数返回对象要加圆括号”。

#### 6.2 创建 Supabase 客户端（可选，为后续认证做准备）

> 需要先安装：`npm install @supabase/supabase-js`

创建 `lib/supabase.ts`：

```typescript
import { createClient } from '@supabase/supabase-js'

const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL
const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY

if (!supabaseUrl || !supabaseAnonKey) {
  throw new Error('Missing Supabase environment variables')
}

export const supabase = createClient(supabaseUrl, supabaseAnonKey)
```

**📖 逐行解释**

- `const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL`：从环境变量读配置。因为我们在 `env.d.ts` 里把它标成了 `string | undefined`，所以下面要先判空。
- `if (!supabaseUrl || !supabaseAnonKey) { throw new Error(...) }`：**提前抛错**——少配了环境变量就立刻报清晰的错误。这一步还有个 TS 好处：经过这个判断后，TS 会“收窄”类型——后面 `supabaseUrl` 就从 `string | undefined` 变成确定的 `string`，传给 `createClient` 不再报错（旧版用 `!` 强行断言非空，不如这样安全）。
